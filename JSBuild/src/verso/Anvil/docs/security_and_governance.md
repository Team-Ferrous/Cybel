# Security, Governance & Audit Standards

## 1. Executive Summary

This repository strictly enforces rigorous architectural and safety mandates mapped from aerospace, automotive, and high-reliability frameworks (NASA-STD-8739.8B, DO-178C). These principles guarantee predictable software lifecycles, fault-tolerant interactions, and rigorous verification requirements for both human engineers and AI agents modifying the Anvil ecosystem.

## 2. The Verification Hierarchy

Anvil adopts Design Assurance Levels (DAL) modeled via the RTCA/FAA Avionics Certification. Testing logic and documentation rigor scale proportionally to task impact.

| Component | DAL | Minimum Verification |
|---|---|---|
| SIMD kernels / gradient-critical ops | DAL-A | Independent review, structural rigor, explicit traceability |
| Python training/audit engine | DAL-B | Requirements-based tests, strict structural coverage |
| Config/scripts/tooling | DAL-C | Functional and regression testing |
| Docs/logs | DAL-D | Review |

## 3. Strict Traceability & The System of Record

### 3.1 Unbroken Chain of Evidence
Every requirement mapping follows a precise, unbreakable path:
`Requirement` -> `Design` -> `Code` -> `Test` -> `Verification Result`
*   No dead code is permitted.
*   No untested requirements.
*   Failures must trace back precisely to the source file and line.

### 3.2 The Repository is the Sole Source of Truth
What an agent cannot read in-repo via `saguaro query` *does not exist*.
-   All AI execution logic and architectural pivots must be documented into explicit Markdown plans.
-   Agents are not permitted to mask defects with clipping/clamping or fallback bridges.

## 4. Advanced Agent Protocols

### 4.1 Quantitative Management & Golden Signals
The system enforces process control constraints:
-   Claims of latency or throughput optimizations require a minimum >30 sample baseline benchmark.
-   Root cause analysis claims mandate precise file:line codebase locations.
-   "Golden Signals" (`step_time_avg_s`, `tokens_per_sec`, CPU utilization, peak RSS) are audited relentlessly.

### 4.2 Spec-Interview Pattern
P0/P1 system logic adjustments (DAL-A/B) must follow the **Spec-Interview Pattern**:
1.  An architecture intent `specs/<task>.md` file must be generated detailing constraints, edge cases, and failure modes.
2.  The implementation mandates independent AI reviewer validation via `saguaro agent` runs to eliminate bias, executing a formalized "Interview" before deployment.

### 4.3 Red Team / Self-Correction
Before finalizing commits via `saguaro agent commit`, a mandatory self-correction workflow requires:
1.  Algorithmic complexity validation (flagging O(n²) or worse)
2.  FMEA (Failure Mode and Effects Analysis) and FTA (Fault Tree Analysis) mapping for critical paths.
3.  A thorough security cross-reference against OWASP Top 10 and SOC2 practices.

## 5. Coding Standards

Software changes across the repository follow standardized baselines:
-   **Python:** Versions >3.12, strict PEP-8 linting via `ruff`, universal type hints.
-   **C++:** C++17, OpenMP execution limits, `alignas(64)` memory allocation rules, bare-metal SIMD intrinsics for hot loops.
-   **General Policies:** Output logic blocks generated by Agents must output completely; placeholder content (`// ... rest of code`) is universally rejected.

## 6. Saguaro Sentinel Enforcement

Execution compliance is maintained via the `saguaro verify .` command, ensuring that Level 3 Governance, including semantic drift, is continuously assessed before any branch may merge. Agents failing to run verification check protocols are permanently halted.
