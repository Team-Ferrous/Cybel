The Anvil Engineering Standard (AES)
1. PREAMBLE
1.1 Purpose and Operating Domains
The Anvil Engineering Standard (AES) constitutes the supreme, unified governing rulebook for all code generation, architectural design, auditing, and maintenance operations performed by the Anvil multi-agent system and its semantic intelligence layer, Saguaro. The AES synthesizes the most rigorous aerospace, automotive, functional safety, and software reliability standards into a cohesive framework adapted for autonomous artificial intelligence execution.1 This standard is engineered to yield software artifacts rivaling the reliability of flight-critical avionics while simultaneously accommodating the rapid iteration cycles demanded by modern computational research.4
This document mandates proof-oriented correctness, structural integrity, and architectural determinism across the following multidimensional domains:
* General Software Engineering: Systems programming, API development, backend optimization, and distributed component architectures utilizing C++17/C++23 and Python 3.12+.6
* Deep Learning / Machine Learning: Training loop integrity, gradient stability, mixed-precision numerical controls, and tensor transformations.
* Quantum Computing: Quantum circuit construction, variational quantum algorithms, noise-aware transpilation, classical-quantum boundaries, and quantum error correction.6
* Physics Simulation: Numerical methods, enforcement of continuous conservation laws, Lie symmetries, and Hamiltonian modeling.7
* High-Performance Computing (HPC): SIMD/AVX2/AVX-512 intrinsic utilization, CPU-first compute fallbacks, strict memory alignment, and cache-line discipline.9
1.2 Interpretation of Mandates
The keywords "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted strictly as absolute engineering directives.
* MANDATORY (SHALL/MUST/MUST NOT): Absolute requirements. The Saguaro intelligence layer MUST block any code commit, architectural proposal, or pull request violating these rules unless accompanied by a cryptographic waiver signed by a human lead architecture team.
* ADVISORY (SHOULD/SHOULD NOT): Highly recommended practices. Anvil agents MUST provide comprehensive, mathematically backed written justification in the commit logs if deviating from an advisory rule.
* ASPIRATIONAL (MAY/OPTIONAL): Optional guidance optimized for exploratory edge cases.

3. UNIVERSAL CODING MANDATES
These mandates govern the foundational logic of the Anvil agent, applying across all languages, libraries, and frameworks.
3.1 Traceability and Architecture
* Rule (AES-TRC-1): Every code artifact MUST trace back to at least one explicitly declared requirement, architectural decision, or mathematical specification. Orphan code MUST NOT exist in AAL-0 or AAL-1 environments.4
* Rule (AES-TRC-2): Architectural changes MUST be documented using Architectural Decision Records (ADRs). ADRs MUST contain Context, Decision, Consequences, and Compliance sections.6
* Rationale: Full traceability (requirement → design → code → test → verification result) prevents feature creep and ensures systematic coverage validation, synthesizing NASA NPR 7150.2D and DO-178C avionics certification standards.2 ADRs prevent the "Groundhog Day" anti-pattern where teams endlessly debate resolved architectural choices.6
3.2 Complexity Bounds and Graph Theory
* Rule (AES-CMP-1): The Cyclomatic Complexity (CC) of any single function or method SHALL NOT exceed 10 to limit control flow divergence.6
* Rule (AES-CMP-2): Time complexity in computational hot paths MUST be strictly bounded. Algorithms exhibiting    or worse asymptotic complexity are PROHIBITED in real-time execution loops unless mathematically proven to operate on strictly bounded datasets   .6
* Rule (AES-CMP-3): Graph structures MUST use adjacency lists for sparse graphs to minimize memory utilization, reserving adjacency matrices strictly for dense graphs where edge lookups require    access times.6
* Rationale: High cyclomatic complexity masks defect vectors and increases the difficulty of achieving MC/DC coverage. Bounded time complexity prevents saturation-induced outages during execution. Understanding asymptotic upper bounds guarantees predictable scaling.6
3.3 Error Handling Philosophy
   * Rule (AES-ERR-1): Cleanroom Philosophy: Root causes MUST be fixed mathematically; symptoms SHALL NEVER be masked by ad-hoc clamping or silent exception swallowing.14
   * Rule (AES-ERR-2): Code MUST NOT use catch(...) or equivalent bare except: blocks unless at absolute process boundaries acting as crash supervisors.
   * Rule (AES-ERR-3): All public APIs MUST return errors explicitly. Control flow MUST NOT rely on exception handling for standard operational logic.6
   * Rationale: Explicit error states force the autonomous agent to handle boundary conditions systematically, averting undefined behavior. The Cleanroom methodology relies on defect prevention rather than debugging.15
3.4 Security Baseline
   * Rule (AES-SEC-1): AAL-0 and AAL-1 systems MUST comply with OWASP ASVS (Application Security Verification Standard) Level 3 requirements, which mandate deep architectural analysis, modularization, and defense-in-depth against advanced vulnerabilities.21
   * Rule (AES-SEC-2): Code generation MUST adhere to NIST SP 800-53 Revision 5 control families, specifically SA-15 for secure software development processes and CM-8 for system component inventory generation.23
   * Rationale: A secure application must be modularized and verified against strict cryptographic and data-flow constraints to survive persistent threats.21
4. LANGUAGE-SPECIFIC STANDARDS
4a. Python Standards (Python 3.12+)
Python is restricted primarily to orchestration, high-level machine learning wrappers, and configuration generation.
   * Rule (AES-PY-1): All functions, variables, and class definitions MUST contain exhaustive PEP 484 and PEP 526 type hints. Saguaro MUST configure static analyzers (mypy) to strict mode.
   * Rule (AES-PY-2): Dynamic execution (eval(), exec()) is STRICTLY PROHIBITED in AAL-0/1 code.
   * Rule (AES-PY-3): Concurrency SHALL utilize explicit asyncio patterns. Threading is discouraged due to the Global Interpreter Lock (GIL); multiprocessing MUST be used for CPU-bound tasks.
   * Rationale: Type-hinting provides pseudo-compile-time verification in an interpreted language, eliminating a major vector of runtime type errors.
4b. C/C++ Standards (C++17 / C++23)
C++ constitutes the core of Anvil's high-performance computing, Linux system programming, and low-latency execution capability.
   * Rule (AES-CPP-1): All C++ code SHALL comply with the MISRA C++:2023 guidelines to prevent undefined, unspecified, and implementation-defined behaviors.26 Language extensions and obsolescent features SHALL NOT be used.28
   * Rule (AES-CPP-2): Resource Acquisition Is Initialization (RAII) MUST be used for all resource management. Raw pointers (new/delete, malloc/free) are PROHIBITED unless encapsulated within a low-level custom memory allocator class.6
   * Rule (AES-CPP-3): Move semantics MUST be explicitly defined for custom types to prevent expensive deep copies of large data buffers or network sockets in high-performance backend systems.6
   * Rule (AES-CPP-4): Error handling SHALL utilize std::expected (C++23) to return values or clear error states without unwinding the stack.6
   * Rule (AES-CPP-5): Compliance with SEI CERT C++ coding standards is REQUIRED (e.g., preventing integer wrapping, dereferencing null pointers, and uninitialized memory reads).29
   * Rule (AES-CPP-6): For Linux systems programming, signal handlers MUST be restricted to async-signal-safe functions. Shared data within signal handlers MUST be declared as volatile sig_atomic_t to ensure uninterruptible single-instruction reads/writes.6

// AES-Compliant C++23 Error Handling, Move Semantics, and RAII
#include <expected>
#include <memory>
#include <string>
#include <cstdint>

enum class SimError { OutOfBounds, MemoryAlignmentFault, Timeout };

class PhysicsBuffer {
   std::unique_ptr<double> data;
   size_t size;
public:
   PhysicsBuffer(size_t s) : data(std::make_unique<double>(s)), size(s) {}
   
   // Enforce Move Semantics (AES-CPP-3)
   PhysicsBuffer(PhysicsBuffer&& other) noexcept = default;
   PhysicsBuffer& operator=(PhysicsBuffer&& other) noexcept = default;
   
   // Delete copy constructors to prevent deep copies
   PhysicsBuffer(const PhysicsBuffer&) = delete;
   PhysicsBuffer& operator=(const PhysicsBuffer&) = delete;

   [[nodiscard]] auto get_ptr() const noexcept -> double* { return data.get(); }
};

// Utilize std::expected for zero-overhead error handling (AES-CPP-4)
auto compute_physics_step(const PhysicsBuffer& buffer) -> std::expected<double, SimError> {
   if (!buffer.get_ptr()) {
       return std::unexpected(SimError::OutOfBounds);
   }
   // Verify 64-byte memory alignment for SIMD operations
   if (reinterpret_cast<std::uintptr_t>(buffer.get_ptr()) % 64!= 0) {
       return std::unexpected(SimError::MemoryAlignmentFault);
   }
   return 1.0; // Return valid output scalar
}

5. DOMAIN-SPECIFIC STANDARDS
5a. Deep Learning / ML Standards
Deep learning systems are highly susceptible to silent failures. Anvil MUST enforce rigorous numerical and state checks to prevent the model from learning corrupted distributions.
   * Training Loop Integrity (AES-ML-1): All training loops MUST include non-finite gradient checks (NaN/Inf). If non-finite values are detected, the loop MUST rollback to the previous checkpoint, adjust learning rates dynamically, and log the precise layer of failure.
   * Mixed Precision Stability (AES-ML-2): If FP16/BF16 is utilized, dynamic loss scaling MUST be applied, and FP32 master weights MUST be maintained to prevent catastrophic numerical accumulation errors.
   * Data Pipeline Validation (AES-ML-3): Input tensors MUST be validated for shape, dtype, and distribution bounds prior to entering the forward pass.
   * Model Reproducibility (AES-ML-4): Pseudo-random number generators (PRNGs) MUST be seeded deterministically. Non-deterministic GPU algorithms SHALL BE disabled during AAL-1 evaluation runs.
   * Rationale: Neural networks will silently learn garbage data if NaNs or structural misalignments propagate. Hard assertions on tensors halt cascading failures before compute cycles are wasted.
5b. Quantum Computing Standards
   * Circuit Design Discipline (AES-Q-1): All parameterized circuits (e.g., Variational Quantum Eigensolvers) MUST define parameter bounds. Over-rotation outside of    MUST be normalized at compile time.6
   * Noise-Aware Programming (AES-Q-2): Any algorithm targeting physical hardware MUST include an explicit noise model simulation step (e.g., depolarizing channels, amplitude damping) in the CI/CD pipeline prior to execution on a Quantum Processing Unit (QPU).6
   * Quantum Error Correction (AES-Q-3): Quantum Error Correction (QEC) protocols MUST explicitly distinguish between error mitigation and logical error correction (e.g., Shor code, stabilizers). When implementing the 9-qubit Shor code, the software MUST mathematically verify that the error probability    is suppressed to   .6 Syndrome extraction logic MUST be tested classically prior to deployment.6
   * Rationale: Quantum algorithms fail silently under environmental decoherence. Simulating noise and testing syndrome extraction ensures classical control software operates correctly against quantum entropy.6
5c. Physics Simulation Standards
      * Conservation Law Verification (AES-PHYS-1): Simulations MUST compute global conserved quantities (energy, linear momentum, angular momentum, mass) at specified temporal intervals. Drift beyond a defined floating-point threshold    MUST trigger a solver rollback or system halt.7
      * Symmetry Enforcement (AES-PHYS-2): Lie symmetries and continuous group transformations MUST be structurally preserved in the numerical design. Neural network surrogates MUST enforce these symmetries algebraically (e.g., using Lie Symmetry Nets) rather than relying purely on soft penalty losses in the objective function, which are mathematically insufficient for long-term integration.8
      * Numerical Method Selection (AES-PHYS-3): For Hamiltonian systems, symplectic integrators MUST be used to strictly bound long-term energy drift. Standard forward Euler integration is STRICTLY PROHIBITED for AAL-0/1 orbital or molecular dynamics.6
      * Rationale: Numerical dissipation of conserved quantities renders long-running physics simulations physically invalid. Symplectic and symmetry-aware algorithms ensure bounded global errors and preserve the fundamental geometry of the modeled system.8
5d. High-Performance Computing Standards
         * CPU-First Rule (AES-HPC-1): All highly optimized SIMD or GPU kernels MUST maintain a strictly equivalent, unoptimized scalar CPU fallback. The CPU implementation acts as the absolute correctness oracle during unit testing.
         * Memory Alignment (AES-HPC-2): Data structures subjected to AVX2 or AVX-512 SIMD operations MUST be explicitly aligned to 32-byte or 64-byte boundaries, respectively, using the alignas() specifier.10 Unaligned loads (e.g., _mm256_loadu_ps) are PROHIBITED in performance-critical hot paths due to cache-line boundary crossing penalties.10
         * Cache Discipline (AES-HPC-3): Data access patterns MUST favor Structure-of-Arrays (SoA) over Array-of-Structures (AoS) to maximize cache line utilization, minimize memory traffic, and prevent false sharing across threads.34
         * Rationale: Misaligned memory access crossing cache-line boundaries severely degrades SIMD performance, bottlenecking OLAP and physics workloads.10 A CPU-first baseline prevents hardware vendor-lock and ensures testing viability.

// AES-Compliant HPC Memory Alignment and SIMD (AVX2) Implementation
#include <immintrin.h>
#include <cstddef>
#include <cassert>

constexpr size_t ALIGNMENT = 32;
constexpr size_t VECTOR_SIZE = 8;

// alignas(32) ensures the base pointer does not cross cache-line boundaries
struct alignas(ALIGNMENT) VectorBlock {
   float data;
};

void compute_vector_add(const float* a, const float* b, float* c, size_t n) {
   // Assert alignment boundaries to prevent hardware penalties (AES-HPC-2)
   assert(reinterpret_cast<std::uintptr_t>(a) % ALIGNMENT == 0);
   assert(reinterpret_cast<std::uintptr_t>(b) % ALIGNMENT == 0);
   assert(reinterpret_cast<std::uintptr_t>(c) % ALIGNMENT == 0);

   for (size_t i = 0; i < n; i += VECTOR_SIZE) {
       // Aligned loads replace unaligned _mm256_loadu_ps
       __m256 va = _mm256_load_ps(&a[i]); 
       __m256 vb = _mm256_load_ps(&b[i]);
       __m256 vc = _mm256_add_ps(va, vb);
       _mm256_store_ps(&c[i], vc);        // Aligned store
   }
}

6. VERIFICATION & TESTING FRAMEWORK
Anvil transitions software engineering from a paradigm of heuristic debugging to statistical correctness validation, adopting IBM Cleanroom methodologies 14 and deep software metrics.6
6.1 Test Taxonomy
         * Unit & Integration Tests: Required for all code. Must execute in deterministic, containerized environments.
         * Property-Based Testing: REQUIRED for all mathematical algorithms, parsers, and physics simulations to prove invariants hold over randomized distributions.
         * Fuzz Testing: Network and API boundaries MUST be fuzzed continuously.
         * Benchmark Testing: Hot paths MUST have historical performance baselines to detect degradation.
6.2 Verification per IBM Cleanroom
         * Rule (AES-VER-1): Software MUST be developed incrementally. Each increment is mathematically verified for correctness against its specification (Clear Box testing) prior to execution. Debugging into correctness is PROHIBITED.15
         * Rule (AES-VER-2): Statistical usage testing SHALL be applied to determine Mean Time to Failure (MTTF). Defect masking is PROHIBITED; failures must trace to logical flaws.35
         * Rationale: Eliminating structural bugs at the mathematical level prior to execution vastly reduces the cost of debugging and ensures high reliability for critical systems, as demonstrated by the zero-defect operations of early Cleanroom implementations.15

10. ANTI-PATTERNS REGISTRY
Anvil MUST detect and autonomously reject the following anti-patterns across all domains:
10.1 General Anti-patterns
                  * Silent Exception Swallowing: try {... } catch (...) {} (C++) or except Exception: pass (Python). Rationale: Destroys observability and masks critical state corruption.
                  * Untraceable Code ("Mystery Features"): Code that implements logic not traced to a requirement definition. Rationale: Violates DO-178C structural constraints.4
                  * Shotgun Dependency Updates: Updating dependencies without a lockfile or regression bounds. Rationale: Introduces supply-chain attack vectors.
10.2 Domain-Specific Anti-patterns
                  * ML - Ad-Hoc Clamping: Using torch.clamp() or tf.clip_by_value() without root-cause analysis to hide exploding gradients. Rationale: Masks fundamentally unstable model architectures.
                  * HPC - Unaligned SIMD Loads: Using _mm256_loadu_ps on misaligned data inside tight loops. Rationale: Causes severe processor pipeline stalls, effectively negating the benefits of vectorization.10
                  * Physics - Euler Integration for Orbits: Using 1st-order forward Euler methods for Hamiltonian mechanics. Rationale: Rapidly introduces non-physical energy into the system, violating conservation laws.6
                  * Quantum - Physical Assumptions in Logical Design: Hardcoding qubit coupling maps directly into algorithmic structure without intermediate transpilation logic. Rationale: Breaks hardware portability.6
